      3 ОПИС ВИКОНАНОЇ РОБОТИ
      3.1 Загальні відомості про шаблони (патерни) проєктування
      Патерни (шаблони) проектування — це перевірені часом, типові способи вирішення поширених проблем, що виникають під час розробки програмного забезпечення. Згідно з класифікацією GoF (Gang of Four), патерни поділяються на три основні групи: твірні , структурні та поведінкові. Область їх застосування є досить широкою і включає об'єктно-орієнтоване проектування, проектування архітектури систем, рефакторинг коду, а також розробку різноманітних фреймворків та бібліотек.
      3.2 Опис обраного патерна
      1. Назва патерна
       Mediator (Посередник).
      2. Тип шаблона
       Поведінковий патерн (Behavioral).
      3. Основне призначення
       Головна мета патерну Посередник полягає у зниженні зв'язності між різними компонентами програмної системи. Замість того, щоб об'єкти безпосередньо взаємодіяли один з одним, утворюючи хаотичну мережу зв'язків типу "всі з усіма", патерн пропонує єдиний центр управління — клас-посередник. Це дозволяє впорядкувати архітектуру, перетворивши її на "зірку", де вся складна логіка комунікації та прийняття рішень інкапсульована в одному місці. Як наслідок, компоненти стають ізольованими та незалежними, що значно полегшує їхнє тестування, підтримку та подальшу модифікацію.
      Передумови застосування
       Впровадження цього шаблону є найбільш виправданим у ситуаціях, коли система починає страждати від так званого "спагетті-коду" — тобто має надмірну кількість жорстких та заплутаних залежностей між класами, де зміна одного елемента непередбачувано ламає інші.
      Також Mediator доцільно використовувати при розробці складних користувацьких інтерфейсів, де події одного елемента (наприклад, натискання кнопки) мають каскадно впливати на стан інших (блокування текстових полів, приховування списків тощо). Крім того, патерн ефективно вирішує проблему розмитої логіки: коли правила взаємодії розкидані по багатьох класах, і їх необхідно централізувати. Зрештою, він стає в пригоді, коли виникає необхідність повторного використання компонентів в інших проєктах, що неможливо зробити, якщо вони "намертво" зв'язані з поточним оточенням.
       Переваги
      1. Ізольованість компонентів: Класи більше не знають про існування один одного, що робить їх повністю незалежними та усуває прямі залежності.
      2. Принцип єдиної відповідальності: Об'єкти фокусуються виключно на своєму прямому функціоналі (бізнес-логіці), делегуючи організацію взаємодії Посереднику.
      3. Гнучкість та перевикористання: Відв'язані класи набагато простіше інтегрувати в інші частини системи або виносити в окремі бібліотеки.
      4. Спрощення архітектури: Уся логіка комунікації зібрана в єдиному місці, що значно пришвидшує пошук помилок та внесення архітектурних змін.
       Недоліки
      1. Ризик появи "Божественного об'єкта" (God Object): У великих проєктах клас-посередник може поступово розростися до некерованих розмірів, увібравши в себе занадто багато логіки, що ускладнить його подальшу підтримку.
      2. Єдина точка відмови: Оскільки комунікація всієї системи зав'язана на одному центрі, критична помилка в роботі Посередника може паралізувати взаємодію всіх компонентів.
      3. Надмірне ускладнення (Оверінжиніринг): Використання Медіатора в простих програмах з мінімальною кількістю зв'язків є невиправданим і лише ускладнює читабельність коду без реальної користі.

      3.3 Структура шаблона (патерна)
Опис структурних частин
       Структура патерну Mediator базується на чіткому розподілі ролей між класами для забезпечення слабкого зв'язування. До основних учасників шаблону належать:
      1. Mediator (Інтерфейс Посередника): Визначає єдиний абстрактний інтерфейс для обміну інформацією між компонентами системи. Найчастіше він містить один метод, наприклад notify(sender, event) або send, через який компоненти сповіщають про події.
      2. ConcreteMediator (Конкретний Посередник): Клас, що безпосередньо реалізує інтерфейс Посередника. Він містить посилання на всі керовані компоненти та інкапсулює в собі всю логіку їхньої координації, чітко знаючи, як система має реагувати на ту чи іншу подію.
      3. Component / Colleague (Базовий Компонент): Базовий клас (або інтерфейс) для незалежних деталей системи. Його головна архітектурна властивість полягає в тому, що він зберігає посилання виключно на інтерфейс Посередника, нічого не знаючи про інші компоненти програми.
      4. ConcreteComponent (Конкретні Компоненти): Класи, що містять основну бізнес-логіку застосунку. Вони автономно виконують свою специфічну роботу, а про всі важливі зміни свого стану сповіщають лише свого Посередника.
Опис взаємодії об’єктів
       Головний принцип взаємодії в патерні Mediator — відсутність прямих комунікацій між компонентами. Об'єкти-колеги ніколи не викликають методи один одного напряму.
      Процес взаємодії виглядає наступним чином:
      1. Коли в ConcreteComponent відбувається певна дія (наприклад, користувач натиснув кнопку), цей компонент не намагається самостійно оновити інтерфейс. Замість цього він викликає метод notify() свого Mediator, передаючи посилання на себе та інформацію про подію.
      2. ConcreteMediator перехоплює це сповіщення. Використовуючи власну логіку умов, він розпізнає відправника та тип події.
      3. Далі ConcreteMediator самостійно приймає рішення і почергово викликає необхідні методи в інших ConcreteComponent (наприклад, дає команду очистити текстове поле чи заблокувати іншу кнопку). Таким чином, утворюється топологія комунікації типу "зірка", де компоненти виступають лише як генератори подій та виконавці команд, а центром управління є Посередник.
UML-діаграма з поясненнями

      Рисунок 3.1 – UML-діаграма структури патерну Mediator
      На класичній UML-діаграмі патерну відображено такі архітектурні взаємозв'язки:
      1. Спадкування / Реалізація: Клас ConcreteMediator реалізує абстрактний інтерфейс Mediator. Водночас конкретні компоненти (ConcreteColleague1, ConcreteColleague2) успадковуються від загального абстрактного класу Colleague.
      2.  Асоціація (Керування): Від ConcreteMediator йдуть спрямовані лінії (посилання) до кожного конкретного ConcreteColleague. Це означає, що Посередник "знає" про існування кожного конкретного компонента і може напряму викликати їхні методи.
      3. Агрегація (Зворотний зв'язок): Від базового класу Colleague йде лінія зв'язку до інтерфейсу Mediator. Це є фундаментом слабкого зв'язування: компоненти знають лише про абстракцію посередника, що дозволяє їм працювати з будь-яким класом, який реалізує цей інтерфейс, не прив'язуючись до конкретної конфігурації системи.
3.4 Реалізація шаблона (патерна)
      1. Опис реалізації мовою програмування Python
      Основні етапи:
      Створення базових інтерфейсів 
      На початковому етапі розробки проєкту формується абстрактний фундамент системи:
      Оголошується абстрактний базовий клас Mediator, який виконує роль інтерфейсу. Він визначає єдиний метод notify, необхідний для обміну повідомленнями між компонентами та центром управління.
      Створюється абстрактний клас SmartDevice (Базовий Компонент), від якого успадковуються всі майбутні розумні пристрої. Його головна задача — зберігати посилання на об'єкт посередника.
      Далі реалізуються класи конкретних пристроїв: MotionSensor (датчик руху), Light (освітлення) та Alarm (сигналізація). Кожен із цих класів містить виключно власну бізнес-логіку. Наприклад, метод detect_motion() у класі датчика лише імітує виявлення руху та викликає метод notify у посередника, передаючи йому свій екземпляр та назву події. Пристрої повністю ізольовані й не містять посилань один на одного (Див. додаток В.1).
2. Реалізація конкретних класів 
      Реалізовано метод register_devices(), який приймає екземпляри конкретних пристроїв, зберігає їх у хабі та одночасно передає кожному пристрою посилання на поточний об'єкт хабу.
      Головна логіка маршрутизації та прийняття рішень інкапсульована в перевизначеному методі notify(). Отримавши сигнал про подію (наприклад, рух), хаб аналізує ситуацію та стан системи. Якщо рух виявлено, він автоматично викликає метод увімкнення світла, а далі, залежно від стану змінної is_armed, приймає рішення — активувати сигналізацію чи проігнорувати подію (Див. додаток В.2).
      3. Демонстрація використання 
      У клієнтській частині скрипта моделюється життєвий цикл системи:
      Спочатку створюються незалежні екземпляри пристроїв (MotionSensor, Light, Alarm).
      Створюється об'єкт посередника (SmartHomeHub), після чого пристрої реєструються в ньому за допомогою методу register_devices.
      Демонстрація відбувається у двох симуляціях. У першій симуляції (режим охорони увімкнено) виклик my_sensor.detect_motion() призводить до того, що хаб вмикає світло та сигналізацію. У другій симуляції користувач змінює стан системи, вимикаючи охорону. При повторному спрацюванні того ж датчика руху хаб лише вмикає світло, не активуючи тривогу. Ця демонстрація підтверджує, що зміна поведінки всієї системи досягається шляхом модифікації стану лише одного класу-посередника, без необхідності втручання в програмний код самих пристроїв (Див. додаток В.3).

      Вимоги до архітектури
1. Створи абстрактний інтерфейс для Посередника, який матиме метод для отримання сповіщень про події від компонентів.
2. Створи базовий клас для всіх розумних пристроїв. Кожен пристрій під час свого створення має обов'язково отримувати посилання на Посередника.
3. Реалізуй три конкретні незалежні пристрої: датчик руху (вміє лише фіксувати рух і відправляти сигнал), розумну лампочку (має дії увімкнення та вимкнення) та сигналізацію (має дію активації сирени). Головна умова: ці пристрої не повинні мати жодних прямих зв'язків між собою.

Після цього він має перевірити стан охорони. Якщо охорона активна — дати команду сигналізації увімкнути сирену. Якщо охорона вимкнена — вивести повідомлення, що тривогу скасовано.

  from abc import ABC, abstractmethod

  # Базовий інтерфейс Посередника
  class Mediator(ABC):
    @abstractmethod
    def notify(self, sender, event: str):
        pass

  # Базовий клас для всіх розумних пристроїв
  class SmartDevice(ABC):
    def __init__(self, mediator: Mediator = None):
        self.mediator = mediator

  # Конкретні пристрої (нічого не знають один про одного)

    def detect_motion(self):
        print("[Сенсор] Виявлено рух у кімнаті!")
        self.mediator.notify(self, "motion_detected")

33   class SmartHomeHub(Mediator):
    def __init__(self):
        self.sensor = None
        self.light = None
        self.alarm = None
        self.is_armed = True  # Режим охорони

    # Реєстрація пристроїв та прив'язка їх до Хабу
    def register_devices(self, sensor, light, alarm):
        self.sensor = sensor
        self.sensor.mediator = self
        
        self.light = light
        self.light.mediator = self
        
        self.alarm = alarm
        self.alarm.mediator = self

    # Вся складна логіка взаємодії зосереджена ТУТ
    def notify(self, sender, event: str):
        if event == "motion_detected" and isinstance(sender, MotionSensor):
            print("[Хаб-Посередник] Отримав сигнал. Аналізую ситуацію...")
            self.light.turn_on() # Світло вмикається завжди
            
            if self.is_armed:
                print("[Хаб-Посередник] Система під охороною! Викликаю тривогу!")
                self.alarm.trigger()
            else:
                print("[Хаб-Посередник] Охорона вимкнена. Тривогу скасовано.")
